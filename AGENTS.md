# AGENTS.md

## 项目角色定位

这是一个面向教师的课堂管理辅助 Web 应用，目标是帮助教师在课堂中实现发言均衡分析与智能分组。项目面向 Vibe Coding 学习场景，因此代码应尽量简单、可读、可测试，方便新手理解前后端协作。

## 主要目标

- 让教师可以注册、登录、创建班级。
- 让教师导入学生名单，班级号作为学生入口。
- 支持学生通过“班级号 + 学号”访问只读视图。
- 支持课堂发言计数与热力图展示。
- 支持基于性别与成绩层级的分组算法。
- 支持每周自动归档历史记录。

## 非目标

- 不追求工业级安全性。
- 不追求高并发与复杂分布式部署。
- 不引入重型框架，不做过度抽象。
- 不追求极致性能优化。

## 设计与实现原则

1. 优先保证“能跑通、容易理解、便于迭代”。
2. 所有代码都应有中文注释，重点逻辑要清晰说明。
3. 使用 HTML + CSS + JavaScript + PHP + SQLite 的简洁组合。
4. 每次提交应尽量只完成一个小闭环功能，避免一次性做太多模块。
5. 一个功能完成后，必须对照设计文档里的验收标准进行验证。

## Agent 工作规则

### 1. 任务拆解原则

- 先做“可验证的最小功能”，不要一次性生成大而全的实现。
- 每个任务必须有明确输入、输出与验收条件。
- 如果任务涉及多文件改动，优先保持最小改动、最小依赖。

### 2. 编码风格要求

- HTML：语义化标签，配合 Bootstrap。
- CSS：优先基于 Bootstrap，必要时覆盖样式。
- JavaScript：ES6+ 原生语法，避免复杂框架。
- PHP：面向过程，直接访问 SQLite。
- 缩进：统一使用 2 空格。
- 变量与函数：使用 camelCase。
- CSS 类名：使用 kebab-case。

### 3. 代码生成规则

- 生成的代码应可直接运行，不依赖外部在线服务。
- 关键功能必须包含易懂的中文注释。
- 不鼓励过度封装，优先保证逻辑直观。
- 如果需要数据库操作，优先使用单文件 SQLite，保持项目零配置。

### 4. 验收规则

在提交实现时，必须确认以下几点：

- 页面能打开且不出现明显 JS 错误。
- 数据能够保存到 SQLite。
- 输入无效时有可理解的提示。
- 功能与 DesignGuide.md 的描述一致。
- 如果新增文件，必须说明用途与依赖。

## 推荐开发顺序

### Phase 1：基础骨架

- 目录结构初始化
- `index.php` / 页面入口
- SQLite 初始化脚本
- 基础 CSS 样式与布局

### Phase 2：教师账号体系

- 教师注册
- 教师登录
- 教师登出
- 管理员身份识别

### Phase 3：班级与名单管理

- 创建班级
- 自动生成 6 位班级号
- CSV 导入学生名单
- 学号唯一识别与更新逻辑

### Phase 4：发言均衡模块

- 学生按钮计数
- 热力图排序与颜色映射
- 随机点名逻辑
- 周归档与历史记录

### Phase 5：智能分组模块

- 成绩等级 A/B/C 自动计算
- 阈值调整
- 性别与成绩分层分组算法
- 分组结果导出

### Phase 6：学生端与只读体验

- 输入班级号与学号
- 查看本人发言次数
- 查看本人分组信息

### Phase 7：收尾与文档完善

- 清理代码与注释
- 补充 README 与 DesignGuide 说明
- 做简单测试与回归检查

## Agent 提示词模板

在使用 Copilot 或其他智能体时，建议按以下格式提问：

```text
请基于 ClassroomTools 项目的 DesignGuide.md，完成以下任务：

目标：
- 实现某个具体页面/功能
- 采用 HTML + CSS + JS + PHP + SQLite
- 注释清楚，面向零基础学习者

限制：
- 不使用复杂框架
- 保持功能简单、可读、可调试
- 先实现最小闭环，不做额外扩展

输出要求：
- 给出完整可运行代码
- 解释关键逻辑
- 指出需要验证的点
```

## 结论

本项目的 Agent 目标不是“零误差完成所有功能”，而是“在最小成本下，帮助新手完成一个可观、可验证、可学习的课堂管理应用原型”。

在开发时，任何新功能都应该先服务于“教师端基础闭环”，再逐步扩展到发言统计、归档历史与智能分组。
